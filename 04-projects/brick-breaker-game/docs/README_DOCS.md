# 문서 사용 가이드

## 📄 문서 종류 및 용도

### 1. PROGRESS.md
**용도**: 전체 프로젝트 개발 과정의 큰 흐름을 기록 (프로젝트 일기)

**업데이트 시점**:
- ✅ 큰 단계(Stage) 완료 시
- ✅ 새로운 기능 추가 완료 시
- ✅ PR 병합 후

**내용**:
- 각 개발 단계별 목표 및 결과
- 완료된 작업 체크리스트
- 다음 할 일 목록
- 프로젝트 전반적인 상태

**특징**:
- 간결한 요약 위주
- 비개발자도 이해 가능한 수준
- "무엇을 했는가"에 집중

---

### 2. REFACTORING_TODO.md
**용도**: 리팩토링 작업의 상세 기록 (기술 노트)

**업데이트 시점**:
- ✅ 리팩토링 단계 시작 시 (목표 및 계획 작성)
- ✅ 리팩토링 진행 중 (상세 구현 내용 추가)
- ✅ 리팩토링 완료 시 (결과 및 해결한 이슈 정리)
- ✅ 설계 결정 사항 발생 시 (왜 이렇게 했는지 기록)

**내용**:
- 리팩토링 TODO 리스트
- 각 Stage별 상세 진행 상황
- 기술적 이슈 및 해결 방법
- 설계 결정 사항 및 이유
- 코드 예시 (Before/After)
- Git 작업 내역

**특징**:
- 기술적 세부사항 포함
- "왜 이렇게 했는가" 중심
- 개발자를 위한 상세 기록
- 나중에 참고하기 위한 아카이브

---

## 🔄 업데이트 예시

### PROGRESS.md 업데이트 예시
```markdown
### Stage 19 (진행 중): 게임 시스템 리팩토링
- **목표**: GameState 추출 및 EffectManager 클래스 분리
- **현재 진행 상황**:
  - ✅ gameState 추출 완료
  - 🔄 EffectManager 클래스 분리 예정
```

### REFACTORING_TODO.md 업데이트 예시
```markdown
### 🔄 Stage 19 진행 중 (2025-11-13~): 게임 시스템 리팩토링

#### ✅ 완료: gameState 추출
**방식 결정: 클래스 vs 단순 객체**
- ❌ 클래스 방식 폐기: 복잡함, 싱글톤 이점 없음
- ✅ 단순 객체 선택: 간결함, import 후 즉시 사용

**구현 내용**:
- 속성: score, lives, difficulty, running, paused
- 메서드: isPlaying(), start(), stop(), ...

**설계 원칙**:
- 싱글톤 패턴
- 캡슐화
- 간결성
```

## 🎯 업데이트 원칙

### PROGRESS.md
- 간결하게 (한 단계당 1-2페이지)
- "무엇을", "왜" 중심
- 체크박스 활용

### REFACTORING_TODO.md
- 상세하게 (필요한 만큼 길어도 됨)
- "어떻게", "왜 이렇게" 중심
- 코드 예시 포함

---

### 3. DESIGN_DECISIONS.md ⭐ NEW
**용도**: 설계 결정 및 기술적 논의 기록 (학습 자료)

**업데이트 시점**:
- ✅ 설계 방식을 선택할 때 (A vs B 비교)
- ✅ 트레이드오프 분석이 필요할 때
- ✅ "왜 이 방식이 더 나은가?" 논의할 때
- ✅ 성능/효율성 비교가 필요할 때

**내용**:
- 설계 선택지 비교 (장단점)
- 기술적 논의 및 이론
- 트레이드오프 분석
- 성능/메모리/가독성 고려 사항
- 설계 패턴 선택 근거

**특징**:
- 학습 자료로 활용 가능
- 비교표와 코드 예시 중심
- "왜 A가 B보다 나은가?" 논리적 설명
- 다른 프로젝트에도 참고 가능

**예시**:
- 클래스 vs 단순 객체 비교
- 콜백 초기화 vs 호출 시 전달
- 객체 참조 vs 인덱스 방식
- 2D 배열 vs 1D 배열

**PROGRESS.md** = 프로젝트 일기 (전체 흐름)
- "무엇을 했는가"

**REFACTORING_TODO.md** = 기술 노트 (상세 기록)
- "어떻게 했는가"

- "왜 이렇게 했는가"
